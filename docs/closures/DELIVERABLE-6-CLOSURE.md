## Anti-cheat self-check

**Anti-cheat directive #1 (No duplicating Senator bootstrap code):** ✅ The CLI does NOT duplicate digest assembly. Both call sites — `inprocess_d4.go`'s `BootstrapSenatorRules` (line 232) and `onboard.go`'s `runOnboarding` — call `BuildRepoDigest` on the librarian Client. `TestOnboardingSynthesizesFromSenatorPipeline/AST_BothPathsCallBuildRepoDigest` walks the AST of both files and refuses to pass unless each references `BuildRepoDigest`. Editing either file to inline a duplicate digest assembly will trip the test.

**Anti-cheat directive #2 (Don't commit force-orchestrator's own ONBOARDING.md):** ✅ `.gitignore` adds `/ONBOARDING.md` at the root. The CLI's smoke run produces the file but it is silently filtered from `git status` / `git add`. Operators who explicitly want a sample committed must copy it to a different filename.

## Architectural notes

**Why `BuildRepoDigest` lives on the Librarian Client interface (not as a free function):** Pattern P16 in CLAUDE.md mandates that cross-agent dependencies route through interfaces in `internal/clients/<service>/`. The CLI needs the digest assembly; the SenatorOnboarding task type needs the same digest assembly; both are agent-equivalent consumers of a Librarian-side data primitive. Putting `BuildRepoDigest` on the Client interface is the canonical shape — the CLI gets a `librarian.NewInProcess(db)` and calls the method, exactly as the daemon's Senate agent does inside the SenatorOnboarding task.

**Repo-spec resolution order:** registered-repo-by-name first (via `store.GetRepo`), with a sanity check that `local_path` is readable. Falls back to "is this an existing directory on disk?" only when the registered-repo lookup fails. That ordering means an operator who runs `force onboard force-orchestrator` from anywhere on disk hits the registered-repo path consistently; the disk-path fallback is for unregistered repos (e.g. a freshly-cloned target the operator hasn't yet `add-repo`'d).

**Pure-data digest, deferred rendering:** `RepoDigest` is intentionally pure data — no Markdown, no LLM calls. The CLI renders to Markdown, the SenatorOnboarding LLM-bootstrap path renders to a prompt fragment. Adding a third consumer (e.g. a Slack /summarise command) is a renderer-only change, not another digest re-implementation.

**Pattern P32 (git ops via igit.LogAndRun) compliance:** the disk-only branch (`unregisteredRecentCommits`) routes through `igit.LogAndRun` with an empty `OpContext{}.Repo`. Empty Repo is allowed by P32; the call still appears in `GitOperationLog` when a DB is attached, satisfying the audit's intent (visibility into every git shell-out).

**No new pattern (P38) test added:** the AST assertion is captured directly in `cmd/force/onboard_test.go` via `TestOnboardingSynthesizesFromSenatorPipeline/AST_BothPathsCallBuildRepoDigest`. Adding a new audittools P38 file would be heavier than the spec calls for; the in-package test is the simplest seam that satisfies "AST walk OR direct equality test" from exit criterion #4. If a future deliverable wants a fleet-wide P38 that walks more call sites, the in-package test promotes cleanly.

## Residual

- **Section depth could grow:** the "Key modules" renderer currently outputs one bullet per top-level directory with a generic "see `<dir>/`" pointer. A future iteration could read each directory's first .go file's package comment to enrich the description. Not blocking the closure.
- **Public API surface scan is regex-only:** for non-Go repos (Python / JS / Ruby / etc.) the scanner is mostly inert. Adding language-specific scanners is a follow-up enhancement, not an exit-criterion gap — the spec explicitly says "exported interfaces, HTTP handlers, CLI commands" which the regex scan covers for Go.
- **The 60-symbol cap is conservative for monorepos:** an operator running `force onboard <huge-monorepo>` will see the cap hit. The output reflects the cap; expanding the cap is a one-line change if real-world feedback warrants.

None of the residuals block CLOSED status.
